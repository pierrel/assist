*** #1 Recommended next upgrade: Qwen/Qwen3-14B-AWQ
Drop-in replacement for the current model. Same weight footprint (~9 GB),
same context window, official AWQ from Qwen team. Qwen3-14B matches
Qwen2.5-32B-Base on most benchmarks. Requires vLLM ≥0.10.

#+begin_src bash
.venv/bin/huggingface-cli download Qwen/Qwen3-14B-AWQ
#+end_src

vLLM serve command:
#+begin_src bash
.venv/bin/vllm serve Qwen/Qwen3-14B-AWQ \
  --host 0.0.0.0 \
  --port 8000 \
  --served-model-name Qwen/Qwen3-14B-AWQ \
  --enable-auto-tool-choice \
  --tool-call-parser qwen3_coder \
  --chat-template-kwargs '{"enable_thinking": false}' \
  --max-model-len 40960 \
  --dtype auto \
  --quantization awq_marlin \
  --gpu-memory-utilization 0.92
#+end_src

Note: use =qwen3_coder= not =hermes= for this model.

*** #2 Capability jump: Qwen3-30B-A3B-AWQ (MoE)
Only 3.3B parameters active per token (MoE), so it fits in ~21 GB at 32k
and runs faster than a 30B dense model. BFCL 69.1. No official HuggingFace
AWQ yet — use ModelScope community quantization.

#+begin_src bash
PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True \
VLLM_USE_MODELSCOPE=True \
.venv/bin/vllm serve swift/Qwen3-30B-A3B-AWQ \
  --host 0.0.0.0 \
  --port 8000 \
  --max-model-len 32768 \
  --max-num-seqs 32 \
  --gpu-memory-utilization 0.90 \
  --enable-auto-tool-choice \
  --tool-call-parser qwen3_coder \
  --reasoning-parser deepseek_r1
#+end_src

*** Avoid for now: Qwen3.5-27B and Qwen3.5-35B-A3B
Both use the GDN (Gated DeltaNet) architecture. Open vLLM bug #35743
(filed 2026-03-02, unresolved): CUDA graph capture fails with
=assert num_cache_lines >= batch= in =causal_conv1d_update=. Workaround
requires =--enforce-eager= which disables CUDA graph optimisation.
Revisit when vLLM ≥0.17 stable ships with GDN support.

*** Avoid: Mistral-Small-3.2-24B
Does not fit on a single 24 GB GPU at full precision (~48 GB). The only
community AWQ has broken tool calling. Not viable for this hardware.

** Passwordless sudo for service management
The Makefile targets use =sudo systemctl= over SSH without a TTY, which requires
passwordless sudo. Run this once from the assist project root:

#+begin_src bash
make vllm-setup-sudo
#+end_src

This streams =scripts/setup-vllm-sudo.sh= to the GPU server, which installs
=/etc/sudoers.d/vllm-service= with the required NOPASSWD entries for
=systemctl=, =journalctl=, and =tee= scoped to the =vllm-serve= service.

Verify with:

#+begin_src bash
make vllm-status
#+end_src

** Managed via Makefile
The assist project manages the vLLM service via =make= targets. From =~/src/assist=:

#+begin_src bash
make vllm-setup    # one-time: install, download model, install systemd service
make vllm-start    # start serving (model loads in 60-120s)
make vllm-health   # check if ready
make vllm-logs     # tail logs
make vllm-restart  # restart after config change
#+end_src

See =.dev.env= for =VLLM_*= configuration vars (host, model, context length, etc).

** Health check

* assist Configuration
After switching models, update =.dev.env= in the assist project root:

#+begin_example
ASSIST_MODEL_URL=http://192.168.6.118:8000/v1
ASSIST_MODEL_NAME=Qwen/Qwen2.5-14B-Instruct-AWQ
ASSIST_CONTEXT_LEN=32768
#+end_example

Note: =ASSIST_CONTEXT_LEN= drops from 78000 to 32768 — Qwen2.5's context window.
This is fine; the multi-agent system rarely exceeds 15k tokens per call.
