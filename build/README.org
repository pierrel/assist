* Model Selection
** Current (as of March 2026)
=mistralai/Ministral-3-8B-Instruct-2512= — served by vLLM on the GPU server.

This model is too small for reliable multi-agent tool calling. See [[*Recommended Upgrade]].

** Recommended Upgrade
=Qwen/Qwen2.5-14B-Instruct-AWQ=

*** Why
- Strongest instruction-following and tool-call fidelity available on this hardware
- Fits in 24 GB at AWQ 4-bit: ~8–9 GB weights + ~6 GB KV cache = ~15 GB total
- Qwen2.5 uses the =hermes= tool-call format — first-class vLLM support
- Resolves all six consistent eval failures that Ministral-3B cannot fix

*** VRAM budget
| Model | Format | Weights | KV cache (15k ctx) | Total | Fits? |
|-------+--------+---------+--------------------+-------+-------|
| Ministral-3B | BF16 | ~6 GB | ~2 GB | ~8 GB | ✅ (current) |
| Qwen2.5-7B | BF16 | ~14 GB | ~4 GB | ~18 GB | ✅ (fallback) |
| Qwen2.5-14B AWQ | INT4 | ~8 GB | ~6 GB | ~14 GB | ✅ (recommended) |
| Qwen2.5-14B | BF16 | ~28 GB | — | >24 GB | ❌ |

*** Latency expectations (RTX 3090)
| Model | Typical call (8k ctx) | Notes |
|-------+-----------------------+-------|
| Ministral-3B | 2–10 s | Fast but unreliable |
| Qwen2.5-7B BF16 | 20–35 s | Good balance |
| Qwen2.5-14B AWQ | 40–70 s | Best quality; within 60 s budget at typical contexts |

** Fallback
=Qwen/Qwen2.5-7B-Instruct= in BF16 (~14 GB, no quantization).
Use if 14B latency consistently exceeds 60 s/call under real workloads.

* vLLM Setup
** Install
#+begin_src bash
cd ~/src/serve
mkdir -p vllm && cd vllm
python -m venv .venv
.venv/bin/pip install vllm huggingface_hub
#+end_src

** Download model
#+begin_src bash
.venv/bin/huggingface-cli download Qwen/Qwen2.5-14B-Instruct-AWQ
#+end_src

** Serve (Qwen2.5-14B-AWQ — recommended)
#+begin_src bash
.venv/bin/vllm serve Qwen/Qwen2.5-14B-Instruct-AWQ \
  --host 0.0.0.0 \
  --port 8000 \
  --served-model-name Qwen/Qwen2.5-14B-Instruct-AWQ \
  --enable-auto-tool-choice \
  --tool-call-parser hermes \
  --max-model-len 32768 \
  --dtype auto \
  --quantization awq \
  --gpu-memory-utilization 0.92
#+end_src

Note: =--host 0.0.0.0= is required for LAN access — the default binds to localhost only.
Note: =--dtype auto= is preferred over =float16= with AWQ models.

** Serve (Qwen2.5-7B — fallback)
#+begin_src bash
.venv/bin/vllm serve Qwen/Qwen2.5-7B-Instruct \
  --host 0.0.0.0 \
  --port 8000 \
  --enable-auto-tool-choice \
  --tool-call-parser hermes \
  --max-model-len 32768 \
  --dtype bfloat16 \
  --gpu-memory-utilization 0.90
#+end_src

** Managed via Makefile
The assist project manages the vLLM service via =make= targets. From =~/src/assist=:

#+begin_src bash
make vllm-setup    # one-time: install, download model, install systemd service
make vllm-start    # start serving (model loads in 60-120s)
make vllm-health   # check if ready
make vllm-logs     # tail logs
make vllm-restart  # restart after config change
#+end_src

See =.dev.env= for =VLLM_*= configuration vars (host, model, context length, etc).

** Health check
#+begin_src bash
curl http://192.168.6.118:8000/health
#+end_src

** Logs
#+begin_src bash
journalctl -u vllm-serve -f
#+end_src

* assist Configuration
After switching models, update =.dev.env= in the assist project root:

#+begin_example
ASSIST_MODEL_URL=http://192.168.6.118:8000/v1
ASSIST_MODEL_NAME=Qwen/Qwen2.5-14B-Instruct-AWQ
ASSIST_CONTEXT_LEN=32768
#+end_example

Note: =ASSIST_CONTEXT_LEN= drops from 78000 to 32768 — Qwen2.5's context window.
This is fine; the multi-agent system rarely exceeds 15k tokens per call.
