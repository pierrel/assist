#+Title: Ministral-3-14B Quantization Guide

* Background

** What is quantization?

A language model's weights are stored as floating-point numbers. By default,
most models use BF16 (bfloat16): 16 bits per parameter. A 14B-parameter model
therefore occupies ~28 GB of memory — more than a 24 GB RTX 3090 can hold.

Quantization reduces the number of bits per parameter at the cost of some
precision. The tradeoff:

| Format   | Bits/param | 14B size | Quality loss    |
|----------+------------+----------+-----------------|
| BF16     |         16 | ~28 GB   | none (baseline) |
| FP8      |          8 | ~14 GB   | negligible      |
| INT4     |          4 | ~7–8 GB  | small (~1–3%)   |
| INT3     |          3 | ~5 GB    | noticeable      |

At 4-bit, a 14B model fits easily on the RTX 3090 (7–8 GB weights + KV cache).
In practice, INT4 quality is close enough to BF16 that most benchmarks show
<2% degradation — often imperceptible in real usage.

** What is AWQ / GPTQ W4A16?

Both are INT4 weight quantization methods. The algorithm used here is *GPTQ
W4A16* (4-bit weights, 16-bit activations) via llm-compressor.

The key insight shared by both: not all weights are equally important. Weights
connected to high-activation channels are more sensitive to precision loss.
Both methods use a small *calibration dataset* to identify these channels and
preserve them.

W4A16 means:
- *W4*: weights stored as INT4 (4 bits) — the memory saving
- *A16*: activations kept in BF16 at runtime — no accuracy loss in the forward pass

The result is a model that uses 4x less memory than BF16 but computes in full
16-bit precision where it matters.

** Tooling: llm-compressor

This guide uses *llm-compressor* (=https://github.com/vllm-project/llm-compressor=),
the vLLM project's official quantization library. AutoAWQ (the previous tool)
is deprecated as of 2026 and llm-compressor is its replacement.

The output format is *compressed-tensors*, which vLLM auto-detects and loads
natively — no =--quantization= flag needed when serving.

** Do I need to provide my own calibration dataset?

No. llm-compressor uses =HuggingFaceH4/ultrachat_200k= as its default
calibration dataset (512 samples of chat conversations). This works well
out of the box.

Using domain-specific calibration data can improve quality for specific use
cases. For tool-calling workloads, calibration samples containing function-call
syntax would be marginally better — but at 14B scale the difference is unlikely
to be measurable in real usage.

#+begin_src python
import logging
import re
import time
from torch import nn
from tqdm import tqdm
from transformers import AutoTokenizer, AutoModelForCausalLM
from datasets import load_dataset
from llmcompressor.modifiers.quantization import GPTQModifier
from llmcompressor import oneshot

from pathlib import Path

MODEL_PATH = Path("/root/models/Ministral-3-14b-Instruct-2512")
OUTPUT_DIR = Path("/root/models/Ministral-3-14B-Instruct-2512-W4A16")
NUM_CALIBRATION_SAMPLES = 512
MAX_SEQUENCE_LENGTH = 2048


class QuantizationProgress(logging.Handler):
    """Intercepts llm-compressor log messages to show per-layer progress with ETA."""

    PATTERNS = [
        r'Compressing\s+(\S+)',
        r'Quantizing\s+(\S+)',
        r'module[:\s]+(\S+)',
    ]

    def __init__(self, model, ignore=("lm_head",)):
        super().__init__()
        total = sum(
            1 for name, m in model.named_modules()
            if isinstance(m, nn.Linear)
            and not any(name.endswith(ig) for ig in ignore)
        )
        print(f"Layers to quantize: {total}")
        self.pbar = tqdm(
            total=total,
            desc="Quantizing",
            unit="layer",
            bar_format="{desc}: {percentage:3.0f}%|{bar}| {n}/{total} layers "
                       "[elapsed: {elapsed}, ETA: {remaining}]",
        )
        self.seen = set()
        self.start = time.time()

    def emit(self, record):
        msg = record.getMessage()
        for pattern in self.PATTERNS:
            m = re.search(pattern, msg, re.IGNORECASE)
            if m:
                key = m.group(1)
                if key not in self.seen:
                    self.seen.add(key)
                    self.pbar.update(1)
                return

    def close(self):
        self.pbar.close()
        elapsed = time.time() - self.start
        print(f"Quantization complete in {elapsed / 60:.1f} min")
        super().close()


assert MODEL_PATH.exists(), f"Model not found at {MODEL_PATH} — check the path"

print("Loading tokenizer...")
tokenizer = AutoTokenizer.from_pretrained(MODEL_PATH, local_files_only=True)

print("Loading model...")
model = AutoModelForCausalLM.from_pretrained(MODEL_PATH, local_files_only=True, torch_dtype="auto")

print("Preparing calibration dataset...")
ds = load_dataset("HuggingFaceH4/ultrachat_200k",
                  split=f"train_sft[:{NUM_CALIBRATION_SAMPLES}]")
ds = ds.shuffle(seed=42)

def preprocess(example):
    return {"text": tokenizer.apply_chat_template(
        example["messages"], tokenize=False)}

ds = ds.map(preprocess)

def tokenize(sample):
    return tokenizer(sample["text"], padding=False,
                     max_length=MAX_SEQUENCE_LENGTH,
                     truncation=True, add_special_tokens=False)

ds = ds.map(tokenize, remove_columns=ds.column_names)

recipe = GPTQModifier(
    targets="Linear",
    scheme="W4A16",
    ignore=["lm_head"],
    group_size=128,
)

progress = QuantizationProgress(model)
logging.getLogger("llmcompressor").addHandler(progress)
logging.getLogger("llmcompressor").setLevel(logging.DEBUG)

print("Running GPTQ calibration and quantization...")
oneshot(
    model=model,
    dataset=ds,
    recipe=recipe,
    max_seq_length=MAX_SEQUENCE_LENGTH,
    num_calibration_samples=NUM_CALIBRATION_SAMPLES,
)

progress.close()
logging.getLogger("llmcompressor").removeHandler(progress)

print(f"Saving to {OUTPUT_DIR}...")
model.save_pretrained(OUTPUT_DIR, save_compressed=True)
tokenizer.save_pretrained(OUTPUT_DIR)

print("Done.")
#+end_src

* Quantizing Locally on the RTX 3090

The RTX 3090's 24 GB VRAM cannot hold the full 28 GB BF16 model at once.
llm-compressor handles this with =sequential=True=, which processes one
transformer layer at a time — loading it onto the GPU, calibrating, quantizing,
then offloading before moving to the next. Your 64 GB DDR5 holds the full model
in system memory throughout.

*Expected time: 4–8 hours.* Each of 512 forward passes shuttles ~40 layers
between RAM and GPU. The actual quantization after calibration is fast.

Note: the GPU will be fully occupied and cannot run vLLM at the same time.

* Troubleshooting

** "Architecture resolved as PixtralForConditionalGeneration"
The model config is pointing to a vision model. This means the base model
download was corrupted or the wrong model was used. Re-download and re-run.

** vLLM doesn't load the quantized model
Add =--quantization compressed-tensors= explicitly to the serve command if
vLLM fails to auto-detect the format.

** OOM during calibration
See the "If it runs out of memory" section above. The primary knobs are
=num_calibration_samples=, =max_seq_length=, and =offload_hessians=.

* Optional: Custom Calibration Dataset

To calibrate toward tool-calling quality, replace the ultrachat dataset:

#+begin_src python
# Custom calibration samples in Mistral's tool-call format
calib_texts = [
    "[INST] What is the weather in Paris? [/INST] [TOOL_CALLS] [{\"name\": \"get_weather\", \"arguments\": {\"city\": \"Paris\"}}]",
    "[INST] Search for recent AI news. [/INST] [TOOL_CALLS] [{\"name\": \"search\", \"arguments\": {\"query\": \"AI news 2026\"}}]",
    # add 50–200 more examples
]

from datasets import Dataset
ds = Dataset.from_dict({"text": calib_texts})

def tokenize(sample):
    return tokenizer(sample["text"], padding=False,
                     max_length=MAX_SEQUENCE_LENGTH,
                     truncation=True, add_special_tokens=False)

ds = ds.map(tokenize, remove_columns=ds.column_names)

oneshot(model=model, dataset=ds, recipe=recipe,
        max_seq_length=MAX_SEQUENCE_LENGTH,
        num_calibration_samples=len(calib_texts))
#+end_src

At 14B scale the improvement over ultrachat_200k is unlikely to be measurable,
but it costs nothing extra if you have the examples available.
