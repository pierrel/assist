#+Title: Ministral-3-14B Quantization Guide

* Background

** What is quantization?

A language model's weights are stored as floating-point numbers. By default,
most models use BF16 (bfloat16): 16 bits per parameter. A 14B-parameter model
therefore occupies ~28 GB of memory — more than a 24 GB RTX 3090 can hold.

Quantization reduces the number of bits per parameter at the cost of some
precision. The tradeoff:

| Format   | Bits/param | 14B size | Quality loss    |
|----------+------------+----------+-----------------|
| BF16     |         16 | ~28 GB   | none (baseline) |
| FP8      |          8 | ~14 GB   | negligible      |
| AWQ INT4 |          4 | ~7–8 GB  | small (~1–3%)   |
| INT3     |          3 | ~5 GB    | noticeable      |

At 4-bit, a 14B model fits easily on the RTX 3090 (7–8 GB weights + KV cache).
In practice, AWQ INT4 quality is close enough to BF16 that most benchmarks show
<2% degradation — often imperceptible in real usage.

** What is AWQ?

AWQ (Activation-aware Weight Quantization) is a 4-bit quantization method from
MIT (Lin et al., 2023). It is the best-performing 4-bit method for inference
with vLLM and has first-class support via =--quantization awq_marlin=.

The key insight: not all weights matter equally. Weights connected to
high-activation channels (the ones the model uses most) are more sensitive to
precision loss. Naively rounding everything to 4 bits destroys these channels.

AWQ's process:
1. Run a small *calibration dataset* through the full-precision model
2. Measure which activation channels have the highest magnitudes (the "salient"
   channels)
3. Scale those weight channels up before quantization, then scale activations
   back down at inference time — effectively giving salient weights higher
   representational precision without using more bits
4. Quantize all weights to INT4 with per-group granularity (group size 128)

The result is a 4-bit model that behaves nearly identically to the BF16 original
on the tasks the calibration data represents.

** Do I need to provide my own calibration dataset?

No. AutoAWQ ships with a default calibration set (512 samples from the Pile
dataset) that works well out of the box. You can get a good quantization without
supplying any data.

That said, using domain-specific calibration data can improve quality for your
use case. Since this model will do tool calling and multi-agent reasoning, a
calibration set that includes function-call examples would be slightly better.
This is optional and unlikely to make a measurable difference at 14B scale —
larger models are more robust to calibration choice.

** How long does it take?

On a cloud A100 40 GB:
- Download 14B model weights: ~10–15 min (depends on HuggingFace speed)
- AWQ calibration pass: ~20–30 min
- Quantization + save: ~5–10 min
- Total: *45–60 minutes*

Total cloud cost at ~$1.50/hr: *~$1–2*.

** Step 3: Run the quantization

Save this as =quantize.py= on the instance:

#+begin_src python
from awq import AutoAWQForCausalLM
from transformers import AutoTokenizer

model_id = "mistralai/Ministral-3-14B-Instruct-2512"
output_dir = "Ministral-3-14B-Instruct-2512-AWQ-4bit"

print("Loading tokenizer...")
tokenizer = AutoTokenizer.from_pretrained(model_id)

print("Loading model (this downloads ~28 GB)...")
model = AutoAWQForCausalLM.from_pretrained(
    model_id,
    low_cpu_mem_usage=True,
    torch_dtype="auto",
)

quant_config = {
    "zero_point": True,
    "q_group_size": 128,
    "w_bit": 4,
    "version": "GEMM",
}

print("Running AWQ calibration and quantization...")
model.quantize(tokenizer, quant_config=quant_config)

print(f"Saving to {output_dir}...")
model.save_quantized(output_dir)
tokenizer.save_pretrained(output_dir)

print("Done.")
#+end_src

Run it:

#+begin_src bash
python quantize.py
#+end_src

The calibration pass uses the default Pile dataset (512 samples). You will see
progress bars for each transformer layer — there are ~40 layers, so expect ~30
minutes.

Note: =awq_marlin= is vLLM's optimised AWQ kernel for Ampere GPUs (RTX 3090).
It is faster than plain =awq= at inference time.

* Troubleshooting

** "Architecture resolved as PixtralForConditionalGeneration"
The model config is pointing to a vision model. This means either the base model
download was corrupted, or the quantizer copied config from the wrong model.
Re-download the base model and re-run.

** "Quantization method in config does not match awq_marlin"
The saved model uses =compressed-tensors= format (Neural Magic's tool) rather than
AutoAWQ format. Drop =--quantization awq_marlin= and let vLLM auto-detect, or
explicitly pass =--quantization compressed-tensors=. This is what happened with
the =cyankiwi= community model.

** OOM during quantization
If you hit out-of-memory during the calibration pass, reduce the number of
calibration samples:

#+begin_src python
model.quantize(tokenizer, quant_config=quant_config, calib_data="pileval", n_parallel_calib_samples=64)
#+end_src

* Optional: Custom Calibration Dataset

If you want to tune the quantization toward tool-calling quality, replace the
default calibration data with function-call examples:

#+begin_src python
# Custom calibration samples — tool-call format
calib_data = [
    "[INST] What is the weather in Paris? [/INST] [TOOL_CALLS] [{\"name\": \"get_weather\", \"arguments\": {\"city\": \"Paris\"}}]",
    "[INST] Search for recent news about AI. [/INST] [TOOL_CALLS] [{\"name\": \"search\", \"arguments\": {\"query\": \"AI news 2026\"}}]",
    # ... add 50–200 more examples in your model's chat format
]

model.quantize(tokenizer, quant_config=quant_config, calib_data=calib_data)
#+end_src

At 14B scale the improvement is unlikely to be significant, but it costs nothing
extra if you have the examples available.
