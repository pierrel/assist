** TODO Tool-call mocking system for evals (record-and-replay cassettes)
*Motivation.*  Eval runs default to real tool calls, which couples
their pass/fail to live network reliability + provider-side response
drift.  Concrete pain points:
- 2026-05-16 PR #89 (Layer 3 eviction) —
  =test_research_multi_turn_token_regression.py= timed out at 702s
  with a stack trace showing =DDGS_*= worker threads stuck on
  =work_queue.get(block=True)= waiting on real DuckDuckGo searches
  that never returned.  0 HTTP requests visible in capture.
- 2026-05-16 PR #101 (context overhaul) — same eval timed out at 900s;
  deeper langgraph chain stuck on =subagent.invoke → runner.tick →
  concurrent.futures.wait=, multiple idle DDGS workers in the executor.
- DDG result drift: a "President of France" query returns different
  ordering / different bodies week-to-week, making any eval that pins
  response shape brittle.

A point fix for DDGS (timeout wrapper / subprocess kill / Tavily swap)
only addresses one tool.  Every other network tool (=read_url=, any
future API integration) has the same exposure.  Mocking at the
tool-call layer is the architecturally right move.

*Design — cassette-style record-and-replay.*  Pattern after
=pytest-recording= / =vcrpy= but at the *tool* layer (not HTTP layer),
because our tools sit above the network and the tool return value is
what assist actually consumes:

1. *Cassette format.*  One JSON file per test, e.g.
   =edd/eval/cassettes/test_research_multi_turn_token_regression.json=.
   Top-level dict keyed by =(tool_name, args_hash)=.  Value is the
   serialized tool return (string, dict, or list — whatever the tool
   normally returns).  Args hash = stable sha256 of
   =json.dumps(args, sort_keys=True)= so the same call with the same
   args replays deterministically.
2. *Replay path (default).*  A pytest fixture (autouse for
   =edd/eval/=) wraps every tool registered with the agent.  On call:
   look up =(tool_name, args_hash)= in the cassette.  Found →
   return the recorded value.  Missing → fall through to the real
   tool call, save the response into the cassette before returning.
3. *Record-from-scratch mode.*  Env var (=ASSIST_EVAL_RECORD=1=) or
   pytest marker (=@pytest.mark.eval_record=).  Deletes the existing
   cassette and re-records from live calls.  Run manually when an
   eval's expected behavior legitimately changes (e.g. a tool's
   schema evolved).
4. *Append mode (default for missing entries).*  Default behavior when
   ASSIST_EVAL_RECORD is unset: replay matches, record misses.  This
   means a brand-new test's first run records cassettes for it; second
   run replays.  Net behavior: cassettes grow over time but never
   block the test.
5. *Strict mode.*  Env var (=ASSIST_EVAL_STRICT=1=) flips missing
   entries from "fall through and record" to "fail loudly."  Use this
   in CI so an unintended live tool call (e.g. a new tool call
   introduced by an agent-prompt change) fails the test instead of
   silently hitting the network.
6. *Scope.*  Per-test cassette (not per-test-class or per-test-file)
   — keeps blast radius small when a single test's prompt changes.
   Cassette path derives from pytest's =request.node.nodeid=.
7. *Time-sensitive determinism.*  Mock the agent's
   =get_current_date()= analog (currently
   =assist.promptable.base_prompt_for= injects =datetime.now()=) to
   a frozen value during eval runs.  Cassettes assume a known wall
   clock to be reproducible.

*Implementation sketch:*
- =edd/eval/conftest.py= — autouse fixture
  =mock_tool_calls(request, tmp_path)= that installs the
  wrap-and-replay shim on every tool the agent registers.  Uses
  =unittest.mock.patch.object= per tool, or registers via a
  middleware that intercepts =wrap_tool_call= (likely cleaner —
  matches the existing middleware pattern in
  =assist/middleware/=).
- =edd/eval/_cassette.py= — load / save / hash logic.  Pure
  functions, no pytest knowledge.
- =edd/eval/cassettes/= — git-tracked JSON cassettes.  PR-time
  review of cassette diffs is the human signal that an eval's
  expected upstream-response shape changed.
- pytest CLI: =--eval-record= (record all), =--eval-strict= (fail
  on miss), default = replay-with-record-on-miss.

*Why middleware (not monkeypatch) for the shim:*
The agent registers tools via =deepagents.create_deep_agent=.  Tools
are invoked through the =wrap_tool_call= chain we've already added
several middlewares to (=OutputSanitizationMiddleware=,
=GitPushBlockerMiddleware=, etc.).  A
=CassetteReplayMiddleware= slots in at that layer with no special
import-time knowledge, and works for both sync (=wrap_tool_call=)
and async (=awrap_tool_call=) paths.  It also naturally handles
subagent tool calls — they go through the same chain.

*Verification path:*
- Pick one eval, e.g. =eval_large_tool_results.py= (already mocks
  DDGS via =unittest.mock.patch('ddgs.DDGS.text', mock_ddg_search)=

- =scripts/run-evals.sh edd/eval/= — should run faster and
  deterministically after migration.
- Add a wiring test (=tests/test_cassette_replay.py=) that
  constructs a 2-call sequence: first records, second replays,

*Replaces / makes obsolete:*

*Open questions for the implementing session:*
1. Cassette diff readability for large tool results — JSON-pretty
   helps but a 50 KB =read_url= response is hard to diff.  Maybe
   gzip + base64 + a sidecar hash file?  Or accept the diff noise
   and rely on PR review judgment.
2. Tool-result drift detection — when running in append mode,
   compare the live response to the cassette and warn if they
   differ structurally (different keys / different list lengths).
   Out of scope for v1; potential v2 feature.
3. How to handle non-deterministic tool args (e.g., a tool that
   uses a random ID).  Cassette miss → real call → mismatched
   args every time → cassette never reused.  May need
   "args canonicalizer" hooks per tool, or accept that some tools
   bypass the cassette.

** TODO Refresh sandbox container TTL on activity (assist-side)
The sandbox container's PID 1 is currently =sleep 3600=
(=dockerfiles/Dockerfile.sandbox:71=), so any thread that runs longer
than an hour loses its container mid-flight.  Symptom:
=SandboxContainerLostError= raised from =DockerSandboxBackend.execute=
(=assist/sandbox.py:69-89=); thread is marked errored.  Not theoretical:
the winged-horse-flag thread (2026-05-16, =20260516165139-36e38c0a=)
was killed at sandbox-start + 1h on a research flow that legitimately
needed >60 min.  Docker events confirmed: =container die execDuration=3600
exitCode=0= followed immediately by =container destroy=.

*Direction:* assist-side refresh-on-activity (NOT docker-internal
=sleep $((3600 - elapsed))= games — those bake the TTL into the image
and can't see "is the agent still working" signals).  The TTL clock
should reset whenever the agent makes a tool call.

Open mechanisms (architect/team to pick):
1. Watchdog thread inside =SandboxManager= that bumps a per-container
   =last_seen= on every =get_sandbox_backend= reuse / every
   =DockerSandboxBackend.execute= call; periodically =container.stop=s
   any container idle >N minutes.  Replace PID 1's =sleep 3600= with
   =sleep infinity= + an external kill signal from the watchdog.
2. Swap PID 1 for an assist-ping-driven process: tiny script that waits
   for a Unix socket signal from the assist host, resetting an internal
   timer each ping; expires if no ping in N minutes.
3. Use =container.restart()= / =container.start()= keepalive — feasible
   only if PID 1 is restart-safe (re-mounts, no state).

Constraints:
- Must NOT bake in extra Docker daemon calls *per tool invocation* (cost
  adds up at 100s of calls/thread; option 1 with batched watchdog ticks
  every N seconds is fine, option 1 with a Docker call per
  =execute()= is not).
- Must still self-destruct on truly-orphan threads — the 1h cap exists
  for a reason (abandoned web threads shouldn't pin sandbox containers
  forever).  Idle-timeout should still fire when there's genuinely no
  activity, just based on assist-observed activity, not wall clock.
- Must coexist with the privilege-separation layer (non-root user,
  internal egress network).

See =assist/sandbox.py:74-77= for the existing TTL discussion and
=SandboxContainerLostError= for the current bubble-up behavior.
