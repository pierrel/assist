** TODO Wrap =git= binary in the sandbox image to refuse =push= at the runtime layer
Static =GitPushBlockerMiddleware= catches direct invocations and shell-outs (=bash -c=, =eval=) but cannot stop dynamic command construction inside Python/Perl/Node.  The follow-up is a =/usr/local/bin/git= shim in the sandbox Docker image that errors on =argv[1]=push= and =exec='s the real git for everything else; closes char-code / base64 / file-mediated bypass routes that no static classifier can detect.  Real production risk is mitigated today by the credential-free sandbox (no SSH key, no =.git-credentials= → push to real remotes fails at the credential layer regardless), so this is defence-in-depth, not blocking.  See "Dynamic-construction attack surface" section in [[file:docs/2026-05-07-per-thread-web-git-isolation.org][docs/2026-05-07-per-thread-web-git-isolation.org]].

#+begin_quote
This could be done through the user expert agent. It should:
1. Intercept any new user request (through a middleware) and be asked to pick out:
   1. Any facts the user has shared. These should be stored in a timestamped file in a declarative format
   2. The main topics of the request (emacs, veganism, etc) and save them to another that is also timestamped
2.Insert into the prompt (again, through middleware) anything relevant to the user request before it reaches the model.

All of these kinds of things should be stored in a special "/memories" directory that is global to the agent. There's just 1 memories directory that is not within the main filesystem. No other agent has access to this.

There should be at least two new tools available to:
1. Calculate topic "density" - The rate of occurrence of topics with some time decay component. This should help the user expert agent understand, if there is any ambiguity, what the user could be asking about. For example, if the user often asks about emacs, then the user expert agent could be used to shape the prompt for the main agent/llm so that this is made clear for example if the user simply asks "how do I split a window horizontally" it could be updated to "how do I split a window horizontally.\n# User context\nThe user is probably asking about this in the context of emacs".
2. Pick out facts (from the facts file) relevant to the user prompt, for example if the user is asking for restaurant recommendations, and there's a fact that the user is vegan, then it should be made clear that the user is probably asking for restaurants that provide vegan options.
