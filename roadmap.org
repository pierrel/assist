*** DEFERRED Layer 2 — Per-thread checkpoint retention

PR #88 should close without merging; revisit a delta-epoch-aware Layer 2 if/when =DeltaChannel= exits beta.

** TODO Replace or sandbox =DDGS= (search_internet) — repeated hangs
=assist/tools.py:43= calls =DDGS().text(query, max_results, backend="duckduckgo")=
with no explicit timeout (DDGS default = 5s per engine query).  Two PRs
across the 2026-05-16 work tripped over this:

- *PR #89* (Layer 3 eviction) — =test_research_multi_turn_token_regression.py=
  timed out at 702s with stack trace showing =DDGS_*= worker threads
  stuck on =work_queue.get(block=True)= waiting on real DuckDuckGo
  searches that never returned.  0 HTTP requests visible in capture.
- *PR #101* (this context overhaul) — same eval timed out at 900s with
  a deeper langgraph chain stuck; stack showed a similar pattern
  (subagent.invoke → runner.tick → concurrent.futures.wait), with
  multiple idle DDGS workers in the executor pool.

*Root cause hypothesis* (needs validation):
1. =DDGS().text(...)= uses =wait(futures, timeout=self._timeout)= but
   does NOT cancel futures left in =not_done=.  Hung workers keep
   running in the executor.
2. =DDGS()= instance is GC'd when =search_internet= returns, BUT the
   shared =get_executor()= behavior across instances may keep the
   thread pool alive across calls.
3. The underlying HTTP lib (=primp=, Rust-based) does an uninterruptible
   socket recv that =primp='s own timeout knob may not enforce.
4. langgraph's parent =concurrent.futures.wait()= has no timeout, so
   any hung future hangs the whole graph.

*Three possible fixes, ordered by escalating intrusion:*

1. *Cheapest:* wrap the whole =DDGS().text()= call in a thread-level
   timeout via =concurrent.futures= with =as_completed(timeout=)=, and
   return =[]= on TimeoutError.  Doesn't fix the leaked thread but
   stops it from hanging the caller.
2. *Medium:* run DDGS in a subprocess (subprocess.run with timeout=)
   so a hung primp socket can be SIGKILL'd cleanly.  More process
   overhead per search but bulletproof.
3. *Right answer for prod:* switch to a paid search backend (Tavily,
   Brave Search API, Google CSE) for prod and keep DDGS as a free-
   tier fallback for dev.  Tavily integration exists upstream
   (=langchain_community.tools.tavily_search=); already in
   requirements.txt as =tavily==1.1.0=.

*Verification path:*
- =scripts/run-evals.sh edd/eval/test_research_multi_turn_token_regression.py=
  is the regression that fails today.  Per-test pytest-timeout currently
  600s; bump to a sane bound after the fix lands so the test fails
  cleanly rather than hangs.
- Look for the eval-window discipline note in
  [[file:reference_nightly_eval_window.md]] — schedule any DDGS-heavy
  validation outside 11p–3a per the cron lockout convention.

*Files to touch:* =assist/tools.py:43-56= for the wrap or backend swap,
=requirements.txt= if dropping/adding deps, design doc per the
doc-per-feature rule.
