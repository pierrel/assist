** Bound production =threads.db= growth
Overall plan + history: [[file:docs/2026-05-03-prod-threads-db-growth.org][docs/2026-05-03-prod-threads-db-growth.org]].  Prod DB hit 187 GB on 2026-05-03 because nothing capped thread count, checkpoints per thread, or content per checkpoint.  Four layers, each with its own design doc and TODO below.

*** TODO Layer 2 — Per-thread checkpoint retention
See [[file:docs/2026-05-04-threads-db-layer-2-checkpoint-pruning.org][docs/2026-05-04-threads-db-layer-2-checkpoint-pruning.org]].  Wrapper over =SqliteSaver= that prunes older checkpoints in =put()= so each thread keeps only =RETAIN_LAST=10=.  Bounds per-thread DB footprint even for hour-long research loops.

*** TODO Layer 3 — Tool-result eviction at write time
See [[file:docs/2026-05-04-threads-db-layer-3-tool-result-eviction.org][docs/2026-05-04-threads-db-layer-3-tool-result-eviction.org]].  Extend =ContextAwareToolEvictionMiddleware= to evict large =ToolMessage= contents to =/workspace/large_tool_results/<id>= before the checkpoint write, leaving a stub in the message list.  Caps single-checkpoint size and prevents =read_url= blobs from accumulating across snapshots.  Eval-side equivalent already shipped (commit f9df751).

** TODO Memory-middleware / tool-API trap
Documented as an open question in the skills doc and partly overlaps with the =write_file= recoverability item above.  The deepagents memory middleware presents synthetic placeholders ("(No memory loaded)", "System reminder: File exists but has empty contents") that the small model treats as editable content; combined with =write_file='s refuse-overwrite policy this produces a no-clean-exit loop.  Either fix needs both ends to align.
