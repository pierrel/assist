* Reliability
** Bound production =threads.db= growth
Overall plan + history: [[file:docs/2026-05-03-prod-threads-db-growth.org][docs/2026-05-03-prod-threads-db-growth.org]].  Prod DB hit 187 GB on 2026-05-03 because nothing capped thread count, checkpoints per thread, or content per checkpoint.  Four layers, each with its own design doc and TODO below.

*** DEFERRED Layer 2 — Per-thread checkpoint retention
See [[file:docs/2026-05-04-threads-db-layer-2-checkpoint-pruning.org][docs/2026-05-04-threads-db-layer-2-checkpoint-pruning.org]] (original design) and [[file:docs/2026-05-16-deltachannel-spike.org][docs/2026-05-16-deltachannel-spike.org]] (decision).  *Deferred indefinitely.*  Wrapper over =SqliteSaver= that prunes older checkpoints in =put()= to keep =RETAIN_LAST=10= per thread.  After the DeltaChannel adoption in PR #100 (42× per-thread file-bytes reduction), per-thread DB footprint is bounded enough that row pruning is no longer urgent — and Layer 2's blind row-pruning may unsafely delete snapshot rows that surviving delta rows reference.  PR #88 should close without merging; revisit a delta-epoch-aware Layer 2 if/when =DeltaChannel= exits beta.

*** TODO Layer 3 — Tool-result eviction at write time (snapshot-aware)
See [[file:docs/2026-05-04-threads-db-layer-3-tool-result-eviction.org][docs/2026-05-04-threads-db-layer-3-tool-result-eviction.org]] (original design) and [[file:docs/2026-05-16-deltachannel-spike.org][docs/2026-05-16-deltachannel-spike.org]] (revised plan post-DeltaChannel).  Extend =ContextAwareToolEvictionMiddleware= to evict large =ToolMessage= contents to =/workspace/large_tool_results/<id>= before the checkpoint write, leaving a stub in the message list.  *Held pending PR #89 rewrite:* under DeltaChannel, =checkpoint["channel_values"]["messages"]= is =None= on most rows and =langgraph.channels.delta._DeltaSnapshot(value=[...])= on snapshot rows.  PR #89's current =EvictionSaver= iterates the field as a list and would silently no-op on delta rows + raise on snapshot rows.  Concrete fix: =isinstance(messages, _DeltaSnapshot)= branch — unwrap → evict → rewrap; =None= → no-op (already evicted upstream); =list= → existing path (writes-call shape).  Plus an eval that asserts eviction fires on a K-aligned step.  Eval-side equivalent already shipped (commit f9df751).

** TODO Clamp =glob= (and sibling fs tools) to the workspace
On 2026-05-16, thread =20260516102611-410cae36= emitted =glob.glob("**/references/**/*", recursive=True)= from =/= inside the sandbox container after =LoopDetection= bounced the research-agent.  Walked =/proc=, =/sys=, and every mounted host path for 18+ minutes at 99% CPU before the container was killed, blocking the whole thread on the Docker exec stream.  Filesystem tools that accept a =path= argument should refuse paths outside the workspace (no =/=, no =..= traversal), and =glob= should reject unbounded-depth patterns that don't anchor inside the workspace.
** TODO Raise total context + smarter context management
llama-server =n_ctx= is currently 96k (down from 131k on 2026-05-16 to keep FA-workspace headroom after the Parler-TTS GPU OOM incident; the lever lives outside this repo in =~/llama-cpp/scripts/serve.sh='s =CTX_SIZE=).  =ContextAwareToolEvictionMiddleware= caps research-agent usable context at ~53k (0.55 × 96k), well below the 78k a single heavy =read_url= can produce.  Two angles:
1. Raise =n_ctx= above 96k once GPU headroom allows — adopting DeltaChannel (see [[file:docs/2026-05-16-deltachannel-spike.org][docs/2026-05-16-deltachannel-spike.org]]) shrinks =threads.db= writes but doesn't change LLM context; the lever here is GPU memory budget after Parler removal + future XTTS load characterization.
2. Smarter eviction.  Today's middleware always evicts the biggest tool result; a long research thread benefits more from /summarizing/ stale context than dropping the most-recent =read_url=.  Semantic compaction is a real lever the current size-based policy can't pull.
