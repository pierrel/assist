#+TITLE: Message events → agent triage → approved actions
#+DATE: <2026-07-01>
#+STATUS: DESIGN — approved-with-changes (Pierre review on PR #164); building

Feature #8. Incoming messages (SMS first) become **events**. A subscription matches senders
by **regexp** and, on a match, fires a turn in that subscription's thread using an
agent-authored **template**. The agent triages the message and may draft a reply; the reply
tool is gated by **deepagents human-in-the-loop** (interrupt), so nothing is sent until the
user approves. Parallels the *schedule* feature (agent-configured per-thread records that
fire a turn), triggered by a matching inbound message instead of a clock.

Product decisions (Pierre): pre-execution gate (via framework HITL); regexp sender routing;
individual events first; reply in v1; the agent does number→name mapping itself. SMS source
= emacsos phone modem; guidance = prose the agent authors into the template.

* Safety: the sandbox contains it; only the reply is gated
Earlier drafts added a deny-by-default "propose-only" enforcer. Dropped (Pierre): the triage
turn runs in the per-turn sandbox, so an injected text steering the agent to do something
bad is contained — worst case, delete the thread (its work is on an isolated thread branch,
never main). The ONE effect that escapes the sandbox is the outbound **reply**; that (not
the whole turn) is what needs a gate, and deepagents already provides one:
~create_deep_agent(interrupt_on={"send_reply": True})~ pauses before the tool runs. The web
thread uses a durable ~SqliteSaver~ (threads.db), so the paused interrupt survives across
requests and restarts; the user approves later and the graph resumes via
~Command(resume=…)~. No custom propose-tool / pending-store / enforcer — the framework HITL
IS the gate. (The triage agent's spec simply doesn't include host-effect tools like
schedule/subscription editing — those aren't sandbox-contained — so its only gated action is
the reply.)

* Model (v1, post-review)
- *Event*: inbound ~{sender, text}~; assist stamps arrival time. Sender is an opaque string.
- *Subscription* (multiple, per-thread — stored like schedules): ~{id, thread_id,
  sender_regexp, template}~. The **template** is the whole self-instruction the agent authors
  — guidance/rules woven in however the agent likes, with ~{sender}~/~{text}~ slots. NO
  separate "rules" field (Pierre C4). Reuse/parameterize ~assist/schedule/store.py~ (don't
  clone it).
- *Routing*: on an inbound message, match its sender against subscriptions' regexps; the
  FIRST match (by creation order) fires a turn in that subscription's thread with its
  rendered template. No match → the message is recorded + logged, no turn (a catch-all is
  expressible as regexp ~.*~). (First-match is the v1 rule; tell-me if you want all-matches.)
- *Triage turn*: a normal sandboxed agent turn on the subscription's thread. Renders the
  template with ~{sender, text}~ — the agent gets the raw sender STRING (Pierre C3) and does
  any number→name mapping itself (Pierre C6, eval-gated; no deterministic mapping). It may
  call ~send_reply(text)~; that tool is HITL-gated.
- *Approve → send*: ~send_reply~ interrupts; the pending reply surfaces in the thread (web
  UI) with approve/reject. Approve → resume → the tool sends via ~POST /outbound/sms~ to the
  phone (~mmcli~). Reject → resume with a decline (the agent is told; no send).
- *Deferred (named)*: richer actions beyond reply; bundled events; other event types;
  user-also-acts; notifications.

* Skills (Pierre C2)
- *~subscribe-events~ skill*: the agent creates/edits subscriptions (regexp + template),
  mirroring the schedule skill's *shape*. Because the config is prose (template) + a regexp
  (not range-checkable structured fields like a schedule's cadence), it's a softer target —
  eval the SETUP, and the skill **strongly recommends loading the regexp skill** when the
  user gives a complex sender rule.
- *~regexp~ skill* (built alongside): helps the agent author + test a correct sender regexp
  (anchoring, escaping, common phone/shortcode shapes). General-purpose, not SMS-specific.

* Reviewer-carried technical decisions (still apply)
- *Inbound route sync ~def~*; dispatch helper sync ~def~ (pinned by a test); approve/resume
  route + the outbound ~requests.post~ off-loop with explicit ~timeout=~.
- *Durable-persist the message before the 200* (ack = "assist owns this text"); dedup +
  ordering from a ~message_id~-keyed record. ~message_id = sha256(sender‖smsc_ts‖text)~ on
  the phone (not the reused modem index). Same-second identical-text collision = documented
  residual.
- Persistence (inbound record, subscription store, routing) uses one lock + atomic
  tmp+rename (ScheduleStore discipline), off the async loop.
- *Auth*: shared secret ~ASSIST_SMS_SECRET~, ~hmac.compare_digest~, missing header → 401,
  unset → 503 (fail closed), both directions. 0600 ~EnvironmentFile~; placeholders in both
  ~.env.example~.

* HITL protocol (verified against installed langchain)
~HumanInTheLoopMiddleware~ (added by ~create_deep_agent(interrupt_on=…)~): configure
~interrupt_on={"send_reply": {"allowed_decisions": ["approve", "edit", "reject"]}}~. When the
agent calls ~send_reply~, the graph raises an interrupt carrying ~action_requests:
[ActionRequest{action, args, …}]~ (args = the draft) — surfaced in the turn result's
~__interrupt__~. The web detects it, sets a status + shows the draft with approve / edit /
reject. Resume by re-invoking the graph (same runconfig; SqliteSaver restores) with
~Command(resume={"decisions": [{"type": "approve"}]})~ (or ~{"type": "reject", "message":
…}~ / ~{"type": "edit", "edited_action": {…}}~). On approve, ~send_reply~'s body runs (POSTs
the reply to the phone). "edit" lets the user tweak the draft before it sends.

* Wire contract
Inbound (phone → assist): ~POST /inbound/sms~, header ~X-Assist-SMS-Secret~, ~{message_id,
sender, text}~ → 200 accepted|duplicate (delete) · 400 malformed (delete) · 401/503
(retain). Outbound (assist → phone, on approve): ~POST /outbound/sms~, same header, ~{to,
text}~ → phone ~mmcli~ send. Canonical contract here.

* Queued (v2) — per-sender stacked pending replies + groups (Pierre, 2026-07-02)
Refine the pending-reply model from *one per thread* to *one per SENDER*:
- A pending reply is keyed by sender (or a GROUP of numbers, for a group message). Messages
  from a given sender still accumulate into that sender's single pending reply (today's
  fold-into-one, but per-sender).
- Different senders STACK: several pending approvals coexist, shown separately in the UI —
  one approval per number / set of numbers.
- The proposal carries the target number(s); a group reply goes to all of them and the
  approval shows the recipients.
- Scope is per-sender PER-THREAD: a different thread can independently queue its own set of
  pending replies, even to the SAME senders as another thread. (Weird but accepted — figure
  the cross-thread UX out then.)
- This resolves the catch-all residual below (which supersedes across senders).
- *Implementation tension to decide then:* today's gate is a SINGLE HITL interrupt per thread
  (a paused graph has one interrupt, so one pending max). Per-sender STACKED pendings need
  either per-sender threads (each with its own interrupt — the original pre-regexp routing
  idea) or a per-sender pending-proposal store layered over the turn (closer to the
  propose-tool + store the first design review floated, before HITL simplified it). Groups =
  a thread/proposal keyed on a set of numbers.

* Risks / residuals
- *Number→name is fully on the agent* (Pierre C6) — the biggest eval target; no code fallback
  in v1.
- *Under-acting* — an ambiguous text yielding no reply AND no signal; the template must have
  the agent end by stating its decision; eval-checked.
- *First-match routing* among overlapping subscriptions (above); *message_id same-second
  collision*; *interrupt lost only if threads.db is lost* (durable otherwise); *reply loop*
  bounded by the approval gate.
