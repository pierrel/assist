Mapped to design decisions:
- *"actual compaction, not hard percentages"* → adopt deepagents'
  =SummarizationMiddleware= (LLM-summarization + offload to
  =/conversation_history/{thread_id}.md=); remove our blunt
  =ContextAwareToolEvictionMiddleware=.
- *"not artificially lowering the ceiling"* → restore =CTX_SIZE=
  131072 (from the temporary 98304 drop on 2026-05-16 morning, which
  was driven by the Parler-TTS GPU squat — Parler is now gated off);
  remove our 0.55–0.60 trigger fractions that double-discounted the
  real ceiling.
- *"only doing it when actually necessary while not hitting OOM"*
  → upstream =SummarizationMiddleware= fires at 0.85 fraction
  /OR/ on =ContextOverflowError= fallback (whichever happens first);
  =BadRequestRetryMiddleware= remains as the terminal safety net.

The whole edifice was a /double safety net/ on top of upstream code
we didn't realize was already running:

- Deepagents 0.6.1 ships =create_summarization_middleware= on by
  default in every =create_deep_agent= call (=graph.py:626, 698=).
  Trigger =("fraction", 0.85)=, keep =("fraction", 0.10)=, with
  offload to =/conversation_history/{thread_id}.md=.  This /is/ the
  "actual compaction" the user asked for.
- Deepagents' =FilesystemMiddleware= already evicts tool results
  >20k tokens to =/large_tool_results/<id>= (=filesystem.py:724-786=).
  Our =ContextAwareToolEvictionMiddleware= ran on top of this.
- =BadRequestRetryMiddleware= already truncates messages >20k chars
  on retry attempts ≥1.  Combined with =SummarizationMiddleware='s
  =ContextOverflowError= fallback, the post-overflow story is strong.

* Design — Option 3 (hybrid, subtractive)

The change is almost entirely deletion.  No new middleware; we just
stop fighting upstream defaults.

| Middleware (ordered as deepagents builds the chain) | Before | After |
|---|---|---|
| =TodoListMiddleware= (deepagents builtin) | active | unchanged |
| =FilesystemMiddleware= (deepagents builtin, evicts >20k token tool results) | active | unchanged — this is what was actually doing the per-result eviction work |
| =SubAgentMiddleware= (deepagents builtin) | active | unchanged |
| *=SummarizationMiddleware=* (deepagents builtin, fraction=0.85, LLM-summarization, offloads to =/conversation_history/=) | *active but invisible* | *active and recognized* — the primary compaction path |
| =PatchToolCallsMiddleware= (deepagents builtin) | active | unchanged |
| *Our =ContextAwareToolEvictionMiddleware=* (custom, fraction=0.55–0.60) | *active* | *deleted* — redundant with the above two |
| =BadRequestRetryMiddleware= (with text sanitization + 20k-char truncation) | active | extended to strip ANSI codes (lifted from the deleted middleware) |
| =LoopDetectionMiddleware=, =ThreadQueueMiddleware=, etc. | active | unchanged |
| =AnthropicPromptCachingMiddleware= (deepagents builtin) | active | unchanged |

The new compaction flow:
1. Tool result arrives.  =FilesystemMiddleware= evicts if >20k
   tokens.  Single-result blowups can't push past this cap.
2. Before model call: =SummarizationMiddleware= checks if total
   context > 0.85 × max_input_tokens (= 0.85 × 131072 = ~111k for
   us post-this-PR).  If yes, runs an LLM-summarization pass and
   offloads the cut history.
3. If the LLM call /still/ returns =ContextOverflowError= (e.g. the
   estimator was off), =SummarizationMiddleware= falls back to the
   summarization path on the spot and retries.
4. If that fails too, =BadRequestRetryMiddleware= truncates every
   message to ≤20k chars (≈5k tokens) and retries (max 3 attempts).
   This is the terminal fail-fast layer — if all three retries fail
   the thread surfaces the error per user's "fail-fast over self-
   healing" preference.

The "actually compacted" answer to the user's question is step 2.
The "only when necessary" answer is step 2's high trigger (0.85 of
the /real/ max_input_tokens, not a 0.55–0.60 discount on a hardcoded
128k that didn't match reality).  The "not OOM" answer is the three-
layer cascade (file eviction → summarization → reactive truncation).

* Risks observed

1. *ANSI proactive vs reactive.*  Previously every tool result was
   sanitized in =after_tool=; now sanitization is reactive (after
   first BadRequestError per thread).  Cost: one extra LLM
   round-trip on the first ANSI-bearing tool result.  Mitigation:
   if eval shows this is a noticeable wall-clock penalty, add a
   tiny =OutputSanitizationMiddleware= in a follow-up that only
   does the ANSI regex in =after_tool=.

2. *Summarization-pass cost on Qwen3.6.*  Each compaction is an LLM
   call sending up to 4000 tokens (langchain default).  At ~40 tok/s
   decode, ~10–30s per compaction.  On a long research thread: 2–4
   compactions, +1–2 min wall-clock per thread.  Worth measuring in
   the eval suite; acceptable in exchange for "actual compaction."

3. *Summary quality on a small local model.*  =DEFAULT_SUMMARY_PROMPT=
   is well-structured but written for SOTA models.  Qwen3.6-27B-Q4_K_M
   may produce lossier summaries than we'd like.  Mitigation: the
   offload-to-=/conversation_history/{thread_id}.md= path means the
   agent can =read_file= the full history if a summary is too thin.
   Eval should look for cases where the agent gets stuck after
   compaction.

4. *DeltaChannel + summarization interaction.*  =SummarizationMiddleware=
   writes to =_summarization_event= (private state field) via
   =wrap_model_call=, NOT via =before_model= with =RemoveMessage=.
   Should not interact with =DeltaChannel='s replay path — verified
   by reading both code paths.  Probe in the realistic eval.

5. *Removing our middleware loses proactive ANSI sanitization on
   non-error paths.*  See risk #1; same mitigation.

6. *Removing our middleware loses the failsafe IF =FilesystemMiddleware=
   or =SummarizationMiddleware= disabled in some future config.*
   Both are auto-installed by =create_deep_agent= and not removable
   without a caller override.  All four of our =create_deep_agent=
   sites use the default behavior.  Low risk.

* Decision
Apply the edits in §File-by-file edits.  This is a *subtractive* PR
— no new code, just removing redundancy and a small extension to
BadRequestRetry.  llama-server CTX_SIZE restore is the only knob
that increases capability rather than reducing complexity.
