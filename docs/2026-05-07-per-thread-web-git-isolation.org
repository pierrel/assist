#+TITLE: Per-thread feature branch for web-UI threads
#+DATE: 2026-05-07

* Status
Design.  Roadmap entry: =roadmap.org= "Per-thread feature branch for web-UI threads" under *Reliability*.

* Problem

Two threads created from the web UI both branch off =main= at roughly the same time and both try to land back to =main=.  Whichever thread merges second hits a non-fast-forward error because =main= advanced under it, even when the two threads' changes do not actually conflict.  The user sees a merge failure that has nothing to do with the substance of their work.

The current contract — implicit in =DomainManager= and the web routes — is unclear.  =DomainManager.has_changes_vs_main()= suggests *some* branch-per-thread machinery exists, but not necessarily what the user intends.  The first step of this work is mapping current behavior; the design below assumes the target contract.

* Target contract

1. *Each web-UI thread runs on its own branch off =main=.*  The branch is created at thread-creation time (not lazily on first commit) so the diff baseline is unambiguous from turn 1.

2. *The agent commits to the thread branch after each turn.*  Diffs the user sees in the UI are "this branch vs latest =origin/main=".

3. *Merge to =main= is auto-rebase + fast-forward.*  When the user merges:
   - Fetch =origin/main=.
   - Rebase the thread branch onto =origin/main=.
   - If rebase succeeds (no conflict), fast-forward =main= to the rebased branch.
   - If rebase produces a conflict, abort the rebase, surface the conflict to the user in the web UI, and tell them to ask the agent to resolve it.  The agent has full git read/write access locally and can run =git rebase --continue= after editing conflicting files.

4. *The agent cannot push.*  Push to =origin/main= is exclusively a user-initiated action from the web UI.  The push action runs on the app side, with credentials the agent does not have, and is implemented as a separate backend endpoint triggered by a UI button.

5. *Conflict surface in the UI.*  When merge fails because of a rebase conflict, the UI shows: "Conflict during merge.  The agent can attempt to resolve — type a message to the thread asking it to fix the conflict."  Once resolved, the user re-clicks merge.

* Decisions baked in (from user)

| # | Decision | Choice |
|---+----------+--------|
| 1 | What "merge even if behind, no conflict" means | *Auto-rebase, then fast-forward.*  Thread branch isn't shared, so the rewrite is safe.  =main= stays linear. |
| 2 | When the thread branch syncs against =origin/main= | *Only at merge time.*  Mid-thread rebases are confusing; conflicts surface late but unambiguously. |
| 3 | Agent push permissions | *Agent can run any git command except push.*  Real enforcement boundary, not docs alone. |
| 4 | Push UI affordance | *User-initiated button + dedicated backend endpoint.*  The endpoint owns the credentials. |

* Decisions made during the Phase 1 architect pass

| # | Decision | Choice |
|---+----------+--------|
| 5 | What happens to the AI-generated commit summary on merge | *Preserved via post-rebase squash.*  Sequence: rebase thread branch onto =origin/main=; if clean, =git merge --squash= into local =main=; commit with the AI-generated summary.  Keeps =main= history at one curated commit per thread; rebase handles divergence. |
| 6 | Per-turn host-side push of the thread branch (=DomainManager.sync= → =git_push=) | *Remove.*  No automated pushes anywhere — push to =origin= only happens on the user-initiated =push-main= action.  Trade-off: in-progress threads are local-only until merge, so a deploy-box loss mid-thread loses the thread.  Acceptable per saved =feedback_thread_failure_should_terminate= memory. |
| 7 | Branch naming refinement | =assist/<UTC-timestamp>-<tid-suffix>= where =tid-suffix= is the last 4 hex chars of the thread id.  Prevents collisions if two threads created in the same second. |
| 8 | Post-merge thread state | Thread rolls forward to a *new* =assist/<ts>-<suffix>= branch (preserves current "keep chatting after merge" UX). |
| 9 | Concurrency model | Single in-process =threading.Lock= in =manage/web/state.py=, held by both =merge_thread= and the new =push-main= endpoint.  Web app is one uvicorn worker — sufficient. |
| 10 | =origin/main= advanced between merge and push | Push endpoint returns 409 with explicit "click Merge to Main again, then click Push again" copy.  No silent retry (per saved =feedback_thread_failure_should_terminate= memory). |
| 11 | Push enforcement layers | Both: =GitPushBlockerMiddleware= wraps the agent's =execute= tool (refuses adjacent =git ... push= tokens) AND sandbox stays credentials-free (no SSH key, no git credentials in the container). |
| 12 | Conflict resolution | New =git-conflict= skill teaching Qwen3-Coder the canonical =git status= → resolve markers → =git add= → =git rebase --continue= loop.  Conflict state persists as =merge_conflict.json= in the thread dir; UI shows red banner. |

* Risks logged by the architect

- *Conflict resolution depends on the small model executing a multi-step git workflow.*  Mitigated by the =git-conflict= skill (Group F).  Residual risk: agent runs =git rebase --abort= / =git push --force= to "fix" the rebase — the wrapper from Group C blocks pushes; the skill explicitly forbids =--abort=.
- *Single-process lock breaks down with multiple workers.*  Acceptable for current single-uvicorn deploy; if that changes, swap for an =flock= on a file in the threads root.
- *=origin/main= advanced between merge and push.*  Per Decision 10 the user gets a 409 and re-merges + re-pushes.  Expect at least one user-confused report; copy must be unmistakable.
- *Backward compat.*  Existing threads with =assist/<ts>= (no suffix) keep their names — the suffix only applies to new threads.  No migration.
- *Layout refactor footprint.*  No package moves planned.  The =python -m manage.web= entry point stays valid.  Real-prod smoke after deploy still mandatory (per saved =feedback_smoke_after_refactor=).

* Open design questions (now answered above — kept for traceability)

These need investigation in the codebase before implementation:

1. *Where does the branch get created today, if anywhere?*  =DomainManager.has_changes_vs_main()= implies branching machinery exists.  Map: thread-creation flow, where =DomainManager= is constructed, what branch state is when a thread starts.

2. *Branch naming.*  Thread-id (stable, opaque) is the safe default.  Slug-from-first-message is friendlier but requires knowing the message before the branch exists.  The architect should propose one.

3. *Where does merge happen today?*  Find the existing merge code path.  Determine whether it currently does ff-only, merge commit, or something else, and where the non-ff failure surfaces.

4. *How do we enforce "agent can't push"?*  Two options to evaluate:
   - Wrap the agent's shell-execution tool to refuse =git push= invocations (any form: =push=, =push origin=, =push --force=, etc.).  Cheap, but the agent could still use porcelain that wraps push.
   - Sandbox-level: mount the working tree without push credentials at all (no =~/.git-credentials=, no SSH key reachable from the sandbox).  Stronger but requires sandbox config changes.
   The right choice probably combines both — the agent sandbox should not have credentials *and* the shell wrapper should refuse =git push=.

5. *Push endpoint design.*  HTTP route in the web app.  Auth model — who is allowed to push?  What happens if the push fails (e.g. someone else pushed in the meantime)?  Probably: re-fetch, re-rebase, re-attempt, surface failure to user if still failing.

6. *Concurrency.*  Two web threads merging at almost-the-same moment.  The rebase + push sequence is not atomic against =origin/main=.  Need a serialization story — likely a lock on the merge endpoint, or accept that one of the two will fail and have to retry.

* Phase 1 — Architect responsibilities

The architect (=Plan= subagent) should:
1. Map current per-thread branching behavior end-to-end (thread create → turn → diff → merge), with file:line references.
2. Identify the gap between current behavior and the target contract above.
3. Answer the six open questions above with concrete choices grounded in the codebase.
4. Produce a numbered implementation plan covering: data model changes (if any), web routes (new + modified), git wrapper / sandbox guard for push enforcement, UI changes, evals/smoke plan.
5. Flag any risks or tradeoffs that the user decisions above didn't anticipate.

* Phase 2 — Implementation responsibilities

After the architect's plan is reviewed and redirected as needed:
1. Implement the changes per the plan.
2. Smoke the merge flow locally (clean ff path AND conflict path).
3. Spawn the reviewer subagent on the diff.
4. Address findings.
5. Real-prod smoke after deploy (per saved =feedback_smoke_after_refactor=) — both clean merge and forced-conflict scenarios.

* Out of scope

- Pushes from anywhere except the web UI button.
- Rewriting =DomainManager= beyond what's needed for the contract above.
- Multi-user concurrent editing of the same thread (the existing single-thread-single-user model stays).
- Fancy conflict-resolution UI — text-based "ask the agent to fix it" is the v1.
