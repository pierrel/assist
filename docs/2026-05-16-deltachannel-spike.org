#+TITLE: DeltaChannel spike — can upstream supersede Layers 2 & 3?
#+DATE: 2026-05-16

* State
Spike complete.  Decision: */adopt the upgrade (Case A — pin bumps
only); close PR #88 (Layer 2); keep PR #89 (Layer 3)./*  See §Decision
for the matrix evaluation and the beta-API + Layer-2-interaction
caveats that drove the close-PR-88 call.

* Goal
Decide, with numbers, whether LangGraph 1.2's =DeltaChannel= (shipped
2026-05-12 alongside =deepagents 0.6.1=) delivers enough per-thread
=threads.db= reduction to:

- *scrap* Layer 2 (per-thread checkpoint retention wrapper, PR #88), or
- *keep but loosen* it (e.g. raise =RETAIN_LAST= 10 → 50), or
- *no-go* — close this branch and merge #88/#89 as planned.

The deepagents 0.6 blog claims ≈40× checkpoint-storage reduction
(5.27 GB → 129 MB on a 200-turn session).  This spike validates that
on /our/ workload before we either rewrite or rip out our custom
wrappers.

* Wire-up — Case A (default-on in deepagents 0.6.1)

=langchain.agents.middleware.types.AgentState= (post-bump, langchain
1.3.1) still uses plain =add_messages=:

#+begin_src python
class AgentState(TypedDict, Generic[ResponseT]):
    messages: Required[Annotated[list[AnyMessage], add_messages]]
    jump_to: NotRequired[Annotated[JumpTo | None, EphemeralValue, PrivateStateAttr]]
    structured_response: NotRequired[Annotated[ResponseT, OmitFromInput]]
#+end_src

But =deepagents 0.6.1=' s =graph.py:63-66= defines:

#+begin_src python
class _DeepAgentState(AgentState):
    """AgentState with DeltaChannel on messages to reduce checkpoint growth from O(N²) to O(N)."""
    messages: Required[Annotated[
        list[AnyMessage],
        DeltaChannel(_messages_delta_reducer, snapshot_frequency=50),
    ]]
#+end_src

…and passes =state_schema=_DeepAgentState= to =create_agent= at
=graph.py:776=.  So *every* caller of =create_deep_agent= picks up
=DeltaChannel= automatically once the pin bumps — including all four
call sites in this repo: =assist/agent.py:194, 248, 366= (main /
context / research agents) plus =edd/agent.py:40= (eval-side agent
builder).  No code edits required.

The reducer is =deepagents._messages_reducer._messages_delta_reducer=
(batch-aware version of =add_messages= — handles =reducer(state,
[write1, write2, ...])=).  Plain upstream =add_messages= is NOT
batch-aware and raises =NotImplementedError= when called directly
from =DeltaChannel.update=.  Implication for the spike harness: it
must use deepagents' reducer, not =add_messages=, to model
production wire-up faithfully.

=DeltaChannel='s docstring carries an explicit beta warning:
#+begin_quote
!!! warning "Beta"
    `DeltaChannel` is in beta. The API and on-disk representation may
    change in future releases. Threads written with `DeltaChannel`
    today are expected to remain readable, but the surrounding contract
    (`BaseCheckpointSaver.get_delta_channel_history`, the
    `_DeltaSnapshot` blob shape, the `counters_since_delta_snapshot`
    metadata field) is not yet stable.
#+end_quote

This warning is load-bearing for the decision below.

* Harness
Two complementary measurements on the same branch:

** Primary — synthetic StateGraph (deterministic, fast)
=edd/spike/measure_deltachannel.py= — builds a minimal =StateGraph=
with one node that appends a synthetic 50 KB =ToolMessage= per
super-step.  Runs N ∈ {20, 50, 100} super-steps against a real
=SqliteSaver= and records bytes-on-disk + per-table row counts +
per-table average blob/value sizes every 5 steps.  Reports stats for
*both* =checkpoints= and =writes= tables so the per-table picture
under =DeltaChannel= is auditable.

Why minimal-StateGraph, not =create_deep_agent=:
- Isolates =DeltaChannel='s effect on serialization from any
  middleware variability.
- Runs in seconds without a model or network dependency.
- Reproducible byte counts (= same run = same bytes ± dict ordering).
- =--delta= flag is required to wrap the messages channel — the
  harness does NOT use =deepagents._DeepAgentState=' s default-on
  =DeltaChannel=, so pin version alone doesn't switch modes.

** Secondary — realistic research eval
=edd/eval/test_research_multi_turn_token_regression.py= (already in
tree, 3 sequential research turns through the real research subagent).
Reuse as-is.  Records final DB size on baseline vs spike.  Realism
check — if synthetic shows 30× and this shows 3×, synthetic isn't
representative.

** Eval-window discipline
Synthetic harness lives in =edd/spike/= (NOT picked up by

* Results

Synthetic harness, constant 50 KB =ToolMessage= per super-step, plain
=SqliteSaver=.  N is the number of graph invocations / super-steps
(harness does not perform any extra seed invoke beyond the loop).

Baseline: =langgraph==1.1.6= / =deepagents==0.5.1= / =langchain==1.2.15=.
Spike: =langgraph==1.2.0= / =deepagents==0.6.1= / =langchain==1.3.1=
(deepagents-side =DeltaChannel(_messages_delta_reducer,
snapshot_frequency=K)= on the =messages= channel).

** Headline file-size reduction

| Workload         | Baseline file_bytes | Spike K=10 file_bytes | Reduction | Spike K=50 file_bytes | Reduction |
|------------------+---------------------+-----------------------+-----------+-----------------------+-----------|
| Synthetic N=20   |          31,076,352 |             3,637,248 |      8.5× |             1,101,824 |    28.2×  |
| Synthetic N=50   |         191,803,392 |            16,658,432 |     11.5× |             6,524,928 |    29.4×  |
| Synthetic N=100  |         764,092,416 |            58,605,568 |     13.0× |            18,087,936 |  *42.2×*  |

All numbers above are post-=PRAGMA wal_checkpoint(TRUNCATE)= — the
harness forces a WAL drain into the main DB file before reading
=os.path.getsize=, so the bytes reflect the true on-disk footprint
of committed rows (not just the main page minus uncheckpointed WAL).
An earlier version of these tables reported 4 KB for the N=20 K=50
cell, which was a WAL-mode artifact — that row's data was in the
=.db-wal= sidecar, not the main =.db= file.

*Important caveat:* this is a /synthetic/ workload.  Real prod turns
include variable-size =HumanMessage= + =AIMessage= thinking blocks +
multiple =ToolMessage=s per step, with =ContextAwareToolEvictionMiddleware=
actively shrinking content mid-thread.  42× is the synthetic upper
bound — matching the deepagents 0.6 blog's ~40× claim is expected
because the blog used the same constant-payload synthetic shape.
*The realistic 3-turn research eval (=test_research_multi_turn_token_regression.py=)
was not run as part of this spike;* schedule it post-merge to
validate the per-thread reduction holds on real research traffic
before quoting the 42× number to operators.

** Where the bytes go — per-table breakdown

Earlier versions of this table reported only =checkpoints= row sums
— misleading under DeltaChannel because the per-step deltas live in
the =writes= table.  Below shows both tables.  (Per-table sums are
slightly less than =file_bytes= because of SQLite page overhead.)

| Workload         | Baseline ckpt rows / avg blob | Baseline writes rows / avg value | Spike K=50 ckpt rows / avg blob | Spike K=50 writes rows / avg value |
|------------------+-------------------------------+----------------------------------+---------------------------------+------------------------------------|
| Synthetic N=20   |             60 / 499 KB       |             60 / 16.9 KB         |             60 / 570 B          |             60 / 16.9 KB           |
| Synthetic N=50   |            150 / 1.26 MB      |            150 / 16.9 KB         |            150 / 25.9 KB        |            150 / 16.9 KB           |
| Synthetic N=100  |            300 / 2.53 MB      |            300 / 16.9 KB         |            300 / 42.7 KB        |            300 / 16.9 KB           |

Observations:
- Row counts are identical between baseline and spike (same number of
  super-steps), in both tables.  Same write cardinality.
- =writes= table per-row value is *unchanged* between modes (~16.9 KB
  per row, both).  =DeltaChannel= does not change the writes shape —
  pending channel updates were already stored there.
- =checkpoints= table per-row blob shrinks dramatically under
  =DeltaChannel=.  Baseline blob grows linearly with N (full
  snapshot), spike blob is tiny for non-snapshot rows and only
  inflates on the K-aligned snapshot rows.  At N=100 K=50, average
  checkpoint blob: 2.53 MB → 43 KB (59× reduction at the per-row
  checkpoint level).
- So the file-size win is *almost entirely* from =checkpoints= blob
  shrinkage; =writes= contributes the same constant ~5 MB at N=100
  in both modes.

** Schema-table check

In all three modes the only tables present are
=['checkpoints', 'writes']=.  No new =deltas= table.  =DeltaChannel=
encodes its per-step deltas as =writes= rows (using the existing
=value= column) — it does not introduce a new table the existing
Layer 2 / Layer 3 wrappers don't expect.  That said, PR #89's read/
write paths still need a =_DeltaSnapshot=-aware branch on the
=checkpoints= column (see §Decision).

* Rollback validation
=RollbackRunnable= (=assist/checkpoint_rollback.py:97-136=) calls
=agent.get_state_history(config)= and indexes =target_idx = depth + 1=.
If =DeltaChannel= exposes delta rows as their own checkpoints, the
math may land on a partial state.  Force a =BadRequestError= at
turn K and check rollback for target indices 1 / 5 / 12 (straddling
=snapshot_frequency= boundary, assuming default K=10).

Harness: =edd/spike/probe_rollback_under_deltachannel.py=.  N=15
super-steps with K=10, so depth=12 straddles the snapshot boundary.
File is named =probe_*.py= (not =test_*.py=) so pytest does NOT
collect it during the normal unit suite — beta DeltaChannel imports
are kept out of the default test path.
The "step" column reports LangGraph's internal step counter
(=metadata.get("step")=), which increments per checkpoint write —
about 3 per super-step (pre-step, post-step, end) — so a 15-super-
step run produces history entries with steps up to ~46.

| Boundary case        | Pass? | Notes                                                            |
|----------------------+-------+------------------------------------------------------------------|
| Rollback depth 1     |  yes  | target step 44 (45 msgs) → resumed step has 48 msgs              |
| Rollback depth 5     |  yes  | target step 40 (42 msgs) → resumed step has 45 msgs              |
| Rollback depth 12    |  yes  | target step 33 (34 msgs) → resumed step has 37 msgs; spans K=10  |

Note this test exercises =app.get_state_history(cfg)= shape +
=app.invoke(input, target_config)= resume, the two LangGraph
primitives =RollbackRunnable= depends on (see
=assist/checkpoint_rollback.py:97,131,144=).  A separate probe in the
same file (=probe_invoke_with_rollback=) installs a graph node that
raises =BadRequestError= on its 6th invocation and drives the full
=invoke_with_rollback= path end-to-end against DeltaChannel.  Confirmed:
the rollback fires at depth=0 (one checkpoint back, step 14), the
retry is the 7th node invocation, and the recovered state contains
the full pre-failure history plus the recovery turn.

* Decision

Applying the pre-measurement criteria.  At =K=50= (the deepagents 0.6.1
production default, set in =deepagents/graph.py:66=), N=100 synthetic:

- *R = 764,092,416 / 18,087,936 = 42.2×* → above the 20× scrap-Layer-2
  threshold.
- *Rollback validation passed* at all three probed depths, including
  straddling the snapshot boundary.

*By the matrix, this is a "scrap Layer 2" outcome.*

But two factors temper that:

1. *=DeltaChannel= is explicitly beta.*  Upstream warning quoted under
   §Wire-up.  The on-disk =_DeltaSnapshot= shape and the
   =BaseCheckpointSaver.get_delta_channel_history= contract are not
   yet stable.  Threads written today are expected to remain readable
   in future releases, but adopting an unstable interface inside a
   stop-the-world wrapper (Layer 2 prunes by row, which is precisely
   the contract that may shift) is fragile.

2. *Layer 2 + DeltaChannel may not compose safely.*  Layer 2 prunes
   the oldest checkpoint rows for a thread.  =DeltaChannel='s replay
   semantics walk ancestor writes back to the last full
   =_DeltaSnapshot=.  If Layer 2's pruning happens to delete the
   snapshot row that surviving delta rows reference, replay breaks
   silently (state reconstructed from an incomplete write chain).
   Today's PR #88 wrapper has no awareness of which rows are
   snapshots vs deltas — it just prunes by =(thread_id,
   checkpoint_id DESC LIMIT -1 OFFSET RETAIN_LAST)=.  This is a
   /correctness/ risk, not a performance one.

*Adopt the upgrade; do not adopt Layer 2.*

Concretely:
- *Bump pins* to =deepagents==0.6.1= / =langgraph==1.2.0= /
  =langchain==1.3.1= and coupled minor bumps.  Picks up DeltaChannel
  automatically via =deepagents._DeepAgentState=.  This is what this
  PR does.
- *Close PR #88* (Layer 2 — per-thread retention) without merging.
  The 42.2× file-size reduction (driven by 59× per-blob shrinkage
  on the checkpoints table) makes the per-thread cap much less
  urgent; the Layer 0 thread-count cap (already shipped) bounds the
  fleet to N_THREADS × ~60 MB at N=100 ≈ 6 GB worst-case.  Plus the
  weekly =VACUUM= reclaims any deletion freelist.  Revisit if/when
  =DeltaChannel= stabilizes and we can rewrite Layer 2 to prune by
  delta-epoch instead of by row.
- *Keep PR #89* (Layer 3 — tool-result eviction at write time) *only after
  it is made `_DeltaSnapshot`-aware.*  PR #89's =EvictionSaver= walks
  =checkpoint["channel_values"]["messages"]= as a =list=.  Under
  =DeltaChannel=, that field is either =None= (non-snapshot rows — the
  full messages list lives in the writes chain) or a
  =langgraph.channels.delta._DeltaSnapshot= instance (snapshot rows
  fired every =snapshot_frequency=50= super-steps).  Direct iteration
  raises on snapshot rows; =None= is silently skipped.  So as-shipped,
  PR #89 would only ever evict on the writes path (=put_writes=) and
  /never/ on the snapshot path (=put=), which is the heaviest row to
  evict.  Empirically verified at K=3 N=7 on this branch:
  =channel_values["messages"]= type is =NoneType= on 23 of 24 rows and
  =_DeltaSnapshot= on the snapshot row.  Concretely PR #89 needs:
  =isinstance(messages, _DeltaSnapshot)= unwrap → evict → rewrap;
  =None= → no-op (already evicted upstream); =list= → existing path
  (covers the writes-call shape).  Plus an eval asserting eviction
  actually fires on a K-aligned step.  Layer 3 still earns its place
  conceptually — it shrinks payload before serialization, compounding
  with deltas — but the wiring needs the snapshot-aware branch first.
  Address PR #89's DDGS-mock eval flake in the same follow-up.

* Risks observed
(Risks from the original planning section that have been *resolved*
during the spike are kept here for traceability with their status.)

1. *Case A vs Case B uncertainty.*  /Resolved — Case A./  Confirmed
   empirically: =deepagents 0.6.1= ships =_DeepAgentState= with
   =DeltaChannel(_messages_delta_reducer, snapshot_frequency=50)= on
   the messages channel (=graph.py:63-66=) and passes
   =state_schema=_DeepAgentState= to =create_agent= (=graph.py:776=).
   Every =create_deep_agent= caller in this repo (4 sites)
   automatically picks it up — no =assist/agent.py= edits required.

2. *=DeltaChannel= beta posture.*  /Confirmed beta;/ docstring
   warning quoted under §Wire-up.  Drove the close-#88 decision (we
   don't want a row-pruning wrapper depending on an unstable
   write-chain contract).

3. *Schema drift breaks PR #88/#89 silently.*  /Half-resolved./
   Spike confirmed tables remain =['checkpoints', 'writes']= — no
   new =deltas= table, so PR #88's =EXPECTED_TABLES= guard does
   NOT trip.  BUT empirically =channel_values["messages"]= becomes
   =None= on most rows and =_DeltaSnapshot(value=[...])= on snapshot
   rows; PR #89's =EvictionSaver= iterates that as a list and breaks
   on both shapes.  Hence the hold-#89 decision.

4. *=RollbackRunnable= history depth.*  /Resolved at non-boundary
   depths; boundary-straddling exercised only by Probe 1./
   =probe_invoke_with_rollback= forces a =BadRequestError= on the
   6th node invocation after 5 successful super-steps, and
   =invoke_with_rollback='s default =max_rollback_depth=3= /
   =max_retries_per_step=2= means it rolls back to depth ≤ 2 — well
   inside the K=10 snapshot window, NOT straddling it.  Boundary-
   straddling resume *is* covered by =probe_history_indexing='s
   depth=12 case (which lands one step past the K=10 boundary and
   succeeds), but that probe resumes with a fresh =HumanMessage=
   rather than the =current_input=None= the production rollback path
   uses.  Both paths pass within their respective scopes; a future
   harness that combines mid-step failure + deeper rollback would
   give end-to-end coverage of "production rollback path AND
   straddles snapshot boundary."  For this PR's scope, the two
   probes together are sufficient evidence that no DeltaChannel-
   specific regression breaks rollback semantics.

5. *Inherited DDGS flake.*  /Avoided./  We did not run the realistic
   eval as part of this spike (synthetic numbers + 407 unit tests
   were considered sufficient given the close-#88 / hold-#89
   decision).  Scheduling the realistic eval is now a post-merge
   task.

6. *One-thread bias.*  /Acknowledged./  =DeltaChannel= helps per-
   thread bytes only.  Layer 0 (thread-count cap, already shipped)
   covers the fleet axis regardless of this PR.
