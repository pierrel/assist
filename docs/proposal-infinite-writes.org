* Proposed Solutions

The two compounding root causes — no-clobber ~write_file~ + missing
loop-termination — should be addressed together. Fixing only the tool
removes /this/ trigger; adding a loop guard prevents the next variant
of the same failure mode (e.g. a different tool that returns repeated
errors).

** Solution 1 — Make ~write_file~ recoverable (root cause #1)

The ~write_file~ tool is provided by ~deepagents/middleware/filesystem.py~
and returns the hardcoded error:

#+begin_quote
Cannot write to {file_path} because it already exists. Read and then
make an edit, or write to a new path.
#+end_quote

The phrase "or write to a new path" is exactly the wrong nudge for a
report-writing agent that has already settled on a canonical filename.
The model latches onto it and starts mutating filenames.

Two non-exclusive options:

1. *Override the tool description / error message.* Wrap the deepagents
   ~write_file~ tool so that when the file exists the error reads
   roughly: "File ~{file_path}~ already exists. Use ~edit_file~ to
   modify it, or stop — do NOT pick a different filename." This is a
   ~_custom_tool_descriptions~-style override and a thin tool-wrapper;
   no fork of deepagents needed.

2. *Add an ~overwrite~ argument (default False).* Lets the report
   writer explicitly opt in to overwriting its own canonical file
   when iterating on critique/fact-check feedback. Default-False keeps
   accidental clobbering off.

Doing (1) alone removes the loop trigger we observed. (2) is a small
ergonomic win for the iterate-on-critique flow.

** Solution 2 — ~LoopDetectionMiddleware~ (root cause #2)

A new middleware that fires in ~before_model~ (or ~after_model~) and
inspects the tail of the message history. It triggers a hard stop
when any of these patterns appears:

- Same tool name + same error class ≥ 2 in a row.
- ≥ 3 ~write_file~ calls to distinct but similar paths within a sliding
  window of ~10 messages (filename-mutation signature).
- Same tool + structurally-identical args ≥ 2 in a row, regardless of
  outcome.

On trigger, the middleware:

- Returns a ~Command~ that injects a terminal ~AIMessage~ summarising the
  last *successful* artifact path (read from state) and then ends the
  graph (no further tool calls).
- Logs a ~loop_detected~ event with the offending tool name and pattern
  so we can build an eval around it.

This belongs alongside the other middlewares in
~assist/middleware/~ and would be wired into ~create_agent~,
~create_research_agent~, and ~create_dev_agent~ in ~assist/agent.py~.
It is /defence in depth/: even if Solution 1 covers ~write_file~, this
catches the same anti-pattern in any other tool that grows a
no-clobber semantics later.

** Solution 3 — Canonical-output tracking for the research agent

State-level enforcement of "Only write the report to a single file":

- When the research agent's ~write_file~ first succeeds for a
  ~*.md~/~*.org~ path, record it as ~final_report_path~ in agent state.
- Subsequent ~write_file~ calls to a /different/ path return a
  ~Command~ pointing the model at ~final_report_path~ with a directive
  to use ~edit_file~ (or finish).

This is more invasive than Solutions 1–2 (touches research-agent
state schema), but it is the only one that turns the soft prompt
constraint ("Only write the report to a single file") into a hard
guarantee. Worth doing if Solutions 1+2 still miss edge cases in
evals.

** Solution 4 — Done-phrase termination heuristic

If an ~AIMessage~ contains a "I have completed the research" /
"final report is at" pattern AND the next step the model proposes is
another ~write_file~, drop the tool call and terminate. Cheap to
implement, narrow in scope, somewhat brittle (string-matching). Treat
as a last-resort patch only if 1+2 prove insufficient.

** Recommended combination

Implement *Solution 1* + *Solution 2* together.

- (1) eliminates the specific trigger from the linked transcript by
  redirecting the model toward ~edit_file~ on collision.
- (2) is a general-purpose safety net that bounds blast radius for any
  future stuck-in-loop bug.

Skip (3) and (4) until an eval (replaying or simulating the
swimming-workout transcript) shows that 1+2 are not enough.
