* Problem
The current architecture loses the user channel when work is delegated to a subagent (e.g. dev-agent), because subagents cannot reply to the user directly.  The supervising general agent paraphrases on the way back, and the next user turn starts a fresh dev-agent with no recollection of prior dev-specific context.

We're using a small model (=stelterlab/Qwen3-Coder-30B-A3B-Instruct-AWQ=), so we cannot solve this by making the system prompt larger and routing everything through one heavyweight top-level agent — the model gets confused by long, multi-domain instructions.

* Proposed direction: skills-based re-architecture
Move the dev/research/context-specific instructions out of dedicated subagents and into *skills* that are lazily loaded into a single top-level agent.  The conversation thread lives at the top level (user channel preserved), but the active context only contains the instructions relevant to the current task.

This is essentially Option 1 from [[file:2026-04-25-dev-improvements.org][dev-improvements.org]] ("expand the general agent"), but with skills as the mechanism for keeping the base prompt small.

** Why skills change the calculus
- Subagent boundaries today are already doing skill-like context partitioning — each agent only loads its own system prompt when invoked.  Skills give us that benefit without losing the user channel.
- Base prompt becomes short: "you're an assistant; here are the skills available."  Dev/research/context guidance becomes lazily-loaded content blocks.
- Multi-turn dev work no longer loses user replies because the conversation never leaves the top-level agent.

** Loading model: model-driven by default
Skills follow the standard pattern: the agent sees skill names + descriptions on every turn and decides via tool call (=load_skill("name")=) when to pull the content in.  The trigger logic lives in the *description* the model reads ("Load when working with =.org= files"), not in our code.  This is uniform across skills and matches how Claude Code's skills work.

*** Risk: the trigger decision is still a decision
Skills only help if the model picks the right one.  Qwen3-Coder is reasonable but not great at meta-reasoning ("which skill applies?").  Mitigations that *don't* compromise the model-driven design:
- Keep the skill set small (4–6 skills, non-overlapping descriptions).
- Invest in skill descriptions: write them as triggers ("Load when X appears"), not as topics ("About X").
- Treat each skill's eval as a probe of "can the model decide to load this?"  A skill whose eval keeps failing is signal that either the description needs work or this particular trigger is too subtle for the small model.

*** Fallback: code-side auto-loading (only if model-driven fails)
If a skill's eval shows the model consistently fails to load it when relevant, we *can* add a side channel — middleware that watches tool results / user messages for a regex and auto-injects the skill content.  Nothing in the agent loop prevents this; we control it.  But it's parallel infrastructure that bypasses the model's judgment, so demote it to fallback: only build it for a specific skill after model-driven loading is shown insufficient for that skill.  Don't build it preemptively.

*** Risk: loading is one-way unless we also build unloading
A long session that pivots (dev → research → dev) accumulates skill content in context.  Without an unload path, skills just delay context bloat rather than solving it.  Hook into the existing =ContextAwareToolEvictionMiddleware= so unused skills are evictable, or expose an explicit =unload_skill= tool.

* Implementation sketch
1. *Skills registry* — a small module mapping skill name → (description, markdown content).  The description is what the model sees on every turn; the content is loaded on demand.  Mirrors the existing =deepagents/*.md.j2= prompt files for content shape.
2. *=load_skill= tool* — model-callable.  Injects the named skill's content as a SystemMessage (or appends to the system prompt).  Optional companion =unload_skill=.
3. *Skill manifest in the base prompt* — append a short "Available skills" block to the top-level agent's system prompt: =name: description= for each skill.  This is the only way the model knows skills exist.
4. *Port the dev-agent instructions first* — =deepagents/dev_agent_instructions.md.j2= is the highest-value test case (it's the source of the problem).  Migrate it to a skill, keep the rest as subagents initially.
5. *Validate, then expand* — only collapse =context-agent= / =research-agent= into skills once the pattern is proven on dev work.  They're cheap and isolated as subagents today.

** Subagent statelessness — finding that strengthens the skills case (2026-04-26)
While planning the dev-agent migration we traced the =task= tool runtime in =deepagents/middleware/subagents.py:344-365=:

#+begin_src python
subagent_state = {k: v for k, v in runtime.state.items() if k not in _EXCLUDED_STATE_KEYS}
subagent_state["messages"] = [HumanMessage(content=description)]
result = subagent.invoke(subagent_state)
#+end_src

Every =task= invocation excludes =messages=, =todos=, =structured_response=, =skills_metadata=, =memory_contents= from the parent's state, then overwrites =messages= with a single =HumanMessage= containing the task description.  No =thread_id= is threaded, so the checkpointer can't preserve cross-call state either.

*Implication*: subagents are structurally one-shot.  Within a single thread, calling dev-agent twice (e.g. "make a plan" then "execute the plan") loses everything from the first call.  Today's dev-agent prompt papers over this by making every invocation a complete explore→plan→tests→impl→critique unit — a workaround, not a fix.  Multi-turn dev work (approvals, mid-task clarifications, plan-then-execute) is *architecturally not supported* by the subagent model.

This is a stronger argument for skills than the original "user channel" framing: with a top-level agent loading the =dev= skill, the conversation thread *is* the dev work — multi-turn flows stay in one continuous context.

Loading the guide *when an =.org= path actually appears* (tool result, user message, or surfaced file path) puts the instruction at a much more salient position — plausibly fixes the failing eval rather than just preserving it.

2. Description (what the model sees in the manifest): something like ="org-format: Load when any .org file path appears in tool results or user messages.  Required before responding about org-mode files."=  The description must double as the trigger instruction.
3. Remove the =\{\% include \%\}= and the line-41 instruction from =context_agent.md.j2=.
4. Run the two evals: =test_context_agent::test_includes_org_format_guidance_for_org_files= (currently failing) and =test_no_org_format_guidance_for_non_org_files= (currently passing).

2. *Load.* Concrete instruction: "Call the =read_file= tool with the skill's =SKILL.md= path."

*** Why this is a good first skill (vs jumping straight to dev)
- *Small surface area*: 39 lines of content, no logic, one trigger.  Bug-finding here is fast.
- *Clear binary signal*: the failing eval gives an immediate go/no-go on whether load-on-demand actually helps the small model.
- *Non-overlapping*: doesn't touch dev/research/memory paths, so a regression here is unambiguously about the skills mechanism.
- *Removes always-on prompt cost*: every non-org context-agent run today carries 39 lines of irrelevant guidance.

* Tradeoff vs. dynamic agent swap (Option 2)
- Option 2 is one-day work: pick the right top-level agent (dev vs general) at session start using project indicators.  Captures most of the user-channel-loss benefit when sessions are mono-domain.
- Skills are higher ceiling, ~one-week work.  Wins clearly when users routinely pivot mid-session (dev → research → dev), and unlocks future skills (PR review, debugging, etc.) without adding more agent boundaries.

If we believe pivoting is common — or if we want a foundation for further capability growth — skills are worth the extra investment.  Otherwise Option 2 is the cheaper validation step and could be done first to confirm the user-channel hypothesis before committing.

- Does =deepagents= already expose a hook for injecting content into the system prompt mid-session, or do we need to add one?
- Should skills be allowed to declare tools (e.g. the "research" skill brings =search_internet= and =read_url=)?  If so, the registry needs to track tool bindings, not just text.
- How do skills interact with subagents we keep around (e.g. =context-agent=)?  Probably: skills augment the top-level agent's prompt; subagents remain orthogonal.
- *Memory-middleware / tool-API trap.* The wrote-nothing analysis (see [[file:baselines/2026-04-25-eval-baseline.md::Wrote-nothing investigation][baseline doc]]) surfaced a workflow trap that's independent of the subagent-vs-skills decision: deepagents' memory middleware presents synthetic placeholders ("(No memory loaded)", "System reminder: File exists but has empty contents") that the small model treats as editable content, combining badly with =write_file='s refuse-overwrite policy. The "memory" skill's instructions can document the right sequence (read first, then write to a fresh path, or use =edit_file= with =old_string=""=), but the underlying tool/middleware behavior is worth fixing separately.
