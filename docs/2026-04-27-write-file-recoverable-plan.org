* Problem
=write_file= is no-clobber.  When the model attempts to write to an
existing path, the deepagents filesystem backend returns:

#+begin_quote
Cannot write to {file_path} because it already exists.  Read and then
make an edit, or write to a new path.
#+end_quote

(=deepagents/backends/state.py:254=,
=deepagents/backends/store.py:494,517=,
=deepagents/backends/filesystem.py:367=, =deepagents/backends/sandbox.py:73=)

The phrase "or write to a new path" is the wrong nudge for a small
model that has already settled on a canonical filename: it picks a
mutated name (=final_report.md= → =completed_final_report.md= →
=report_final.md=…) and burns through tool calls.

Loop detection (=LoopDetectionMiddleware= Pattern C) catches the
runaway after a few cycles, but the failure is still real: the agent
either gives up entirely or writes to the wrong file.  Symptoms in the
2026-04-25 baseline:

- =test_memory::test_doesnt_include_tags= — wrote to =/user_cats_info.md=
  instead of =/AGENTS.md= because =AGENTS.md= already existed (with
  middleware-synthetic placeholder content).
- Research-agent transcripts in =threads/20260424061644-1b5485b7/domain/=
  with nine near-identical report files.
- =test_mid_task_clarification= flake during the dev-skill migration
  (write_file collision on =calculator.py= during turn-2 clarification).

Proposal source: =docs/proposal-infinite-writes.org= (Solution 1).

* Goal
Make =write_file= recoverable on collision: when the file exists,
return an error that *positively redirects* the model to
=edit_file=, without nudging it to invent a new filename.  The fix
should be a small, well-scoped middleware that follows existing
patterns (=ReadOnlyEnforcerMiddleware=, =ContextAwareToolEvictionMiddleware=)
and works across all backends (state, store, composite, sandbox).

Non-goals:
- Solution 2 (=overwrite= arg): not in scope.  Adding tool args means
  forking deepagents; out of proportion for what we need.
- Solution 3 (canonical-output state tracking): not in scope.
  Premature unless 1+ proves insufficient.

* Approach
A new middleware =WriteCollisionMiddleware= in =assist/middleware/=
that intercepts =write_file= via =wrap_tool_call=.  After the tool
returns, the middleware inspects the result; if it matches the
collision-error signature, the result is rewritten to a clearer
message before being added to the message list.

** Why middleware (not tool description override)
The proposal lists two implementation paths.  Tool description
override (=_custom_tool_descriptions=) only affects the *tool
description* shown to the model up front — it cannot rewrite the
*runtime error message* the backend returns on collision.  The
filename-mutation trap is triggered by the runtime error, not the
description.  A =wrap_tool_call= middleware sees the actual error and
can rewrite it.  This also keeps the fix backend-agnostic.

** Detection signature
The error string is hard-coded in deepagents in three backend files
(state, store, filesystem) all using the same template:

#+begin_quote
Cannot write to {file_path} because it already exists. ...
#+end_quote

Plus the sandbox backend uses a slightly different shape:

#+begin_quote
Error: File already exists: {repr(path)}
#+end_quote

The middleware will detect both via a single regex: a leading "Cannot
write to" OR "Error: File already exists:".

** New error message
Replace the deepagents text with one designed for a small model:

#+begin_quote
Error: =/path/to/file= already exists.  To modify it, call =read_file=
to see the current content, then call =edit_file= with the exact
=old_string= and =new_string= you want to change.  Do NOT call
=write_file= again with a different filename — the existing file is
the one you should edit.
#+end_quote

Three concrete properties:
1. *Names the existing file.*  No ambiguity about which file.
2. *Points at =edit_file=* with exact next steps (=read_file= first).
   Positive redirection beats a "do not" instruction for a small
   model.
3. *Names the failure mode the small model is prone to* ("Do NOT call
   write_file again with a different filename").  Negative
   reinforcement of the specific bad behaviour.

* Risks and gotchas
- *Regex false positive*: the detection regex must not match
  unrelated errors that happen to contain the substring "Cannot write
  to".  Mitigation: anchor the regex to the start of the message
  ("^Cannot write to ").  Test the false-positive case in the unit
  eval.
- *Sandbox backend shape difference*: the sandbox-specific error
  ("Error: File already exists: ...") needs separate detection.
  Covered by an OR in the regex.
- *Async path*: deepagents has both =sync_write_file= and
  =async_write_file=.  =wrap_tool_call= covers both because it
  intercepts at the request level, not the function level.  Confirmed
  by reading =deepagents/middleware/filesystem.py:805-848=.
- *Other tools that share the wording*: =edit_file= can return
  similar messages on file-not-found, but its error template doesn't
  start with "Cannot write to".  Filtering by =request.tool_call["name"]
  == "write_file"= avoids cross-tool contamination.
- *Subagent paths*: subagents (=context-agent=, =research-agent=)
  inherit middleware lists in =assist/agent.py= but only get the
  subset passed in =[retry_middle, json_validation_mw, tool_name_mw]=.
  We need to pass =write_collision_mw= explicitly to subagents that
  can write — i.e. =research-agent=.  Context-agent has the
  read-only enforcer and never gets to =write_file=.
